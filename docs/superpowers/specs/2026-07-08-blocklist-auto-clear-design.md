# ceph_watchdog: автоматическое снятие blocklist — дизайн

**Дата:** 2026-07-08
**Статус:** одобрено пользователем

## Контекст и находка

Расследование реальных инцидентов на arch03 (2026-07-08, три отдельных случая: 13:13, 15:06, 19:39) показало, что цикл `mds0 caps stale` → `mds0 hung` → `rejected session`, который `ceph_watchdog` уже умеет обнаруживать и на который реагирует перемонтированием, — не вся картина. Логи самого MDS (`podman logs ceph-mds-cephrgw01` на cephrgw01) показывают:

```
log_channel(cluster) log [WRN] : evicting unresponsive client arch03:test (...), after 301-314 seconds
Evicting (and blocklisting) client session ...
```

MDS явно **вытесняет и блокирует** (`ceph osd blocklist`) клиента, если тот не отвечает на heartbeat/cap-recall ~5 минут. Пока запись в blocklist активна, клиент физически не может переподключиться — обычные попытки `umount`/`mount` внутри `ceph_watchdog_recover()` обречены на провал, сколько бы их ни было. Снятие блокировки вручную (`ceph osd blocklist rm <client-addr>` на кластере) подтверждено вживую: `/ceph` монтируется мгновенно сразу после снятия. Без вмешательства запись может висеть до часа.

## Цель

Дать `ceph_watchdog_recover()` возможность самостоятельно снять собственную запись из blocklist кластера, если 5 обычных попыток перемонтирования не сработали, и сделать ещё одну попытку — вместо того чтобы сдаваться и ждать, пока запись истечёт сама (до часа простоя).

## Архитектура

Новый шаг встраивается как **fallback в конце уже существующего цикла** `ceph_watchdog_recover()` (`backup/rclone_backup_unified_v4.0.0.sh`), выполняется только если все 5 обычных попыток `mount` провалились — без изменения самого 5-попыточного цикла.

### Credential

На стороне кластера (cephrgw01, через уже подтверждённый маршрут amar319 → wn75 → cephrgw01 → `podman exec ceph-mon-cephrgw01`) создаётся новый ограниченный клиент:

```
ceph auth get-or-create client.watchdog \
  mon 'allow command "osd blocklist rm", allow command "osd blocklist ls"'
```

Права — только эти две команды через mon, ничего больше (не полный admin, не файловый доступ к CephFS). Пользователь явно согласился, что технически этот credential может снять blocklist **любого** клиента кластера, не только своего — это приемлемо в рамках доверенной внутренней сети.

### Хранение на arch03/04/05

Новые, отдельные от существующих файлы (существующий `/etc/ceph/ceph.conf`/keyring на этих хостах устарел — указывает на несуществующие mon IP 172.17.200.15-17 вместо реальных 172.17.200.85/86/87, — не трогаем и не чиним его, это вне скоупа):

- `/etc/ceph/ceph.watchdog.conf` — минимальный conf с актуальными `mon host = 172.17.200.85,172.17.200.86,172.17.200.87` и правильным `fsid`.
- `/etc/ceph/ceph.watchdog.keyring` — keyring для `client.watchdog`, права на чтение только для root (0600), скрипт и так работает от root.

### Логика в ceph_watchdog_recover()

После существующего цикла `for (( attempt = 1; attempt <= CEPH_WATCHDOG_REMOUNT_ATTEMPTS; attempt++ )); do ... done` (`backup/rclone_backup_unified_v4.0.0.sh:846-865`), перед текущим финальным `log ERROR "не удалось перемонтировать..."` (строка 867), — новый блок:

1. Определить собственные IP-адреса хоста (`ip -4 addr show`, все адреса кроме loopback).
2. Запросить `ceph -c /etc/ceph/ceph.watchdog.conf --keyring /etc/ceph/ceph.watchdog.keyring --connect-timeout 10 osd blocklist ls` (с таймаутом на случай недоступности мониторов).
3. Для каждой строки листинга, чей IP совпадает с одним из собственных, выполнить `osd blocklist rm <точная строка адреса>`.
4. Если хотя бы одна запись была снята — сделать **одну** дополнительную попытку `mount /ceph` + `ceph_watchdog_check` (переиспользуя ту же логику, что уже есть внутри цикла, но не запуская цикл заново).
5. Если эта попытка успешна — залогировать успех через новый явный лог (`ceph_watchdog: снята собственная блокировка, /ceph перемонтирован`) и `return 0`, как и обычный успешный путь.
6. Если снятие блокировки не помогло (записей не было, команда не выполнилась, или мониторы недоступны, или последующий mount всё равно не удался) — переходим к уже существующему поведению без изменений: лог `"не удалось перемонтировать /ceph после N попыток"`, `return 1`.

Никаких изменений в основном 5-попыточном цикле, в `ceph_watchdog_check()`, в детектирующей части `ceph_watchdog()`, во flock-координации с `main()`/`cleanup()` — новый блок целиком находится внутри уже существующей эксклюзивной блокировки восстановления.

## Обработка ошибок

Любая неудача на шагах 1-3 (нет сетевого доступа к mon, keyring отсутствует/повреждён, `ceph` команда зависла/вернула ошибку) не должна прерывать скрипт — просто пропускаем блок и переходим к существующему пути `return 1`, то же самое поведение, что и сейчас без этой фичи. Таймаут на сетевые вызовы `ceph` обязателен (по аналогии с `timeout -k` на `mount`/`umount` в этой же функции) — недоступный монитор не должен зависать неопределённо.

## Тестирование

- Провизия `client.watchdog` и файлов — проверяется вручную на реальном кластере (контроллер выполняет с текущим доступом к cephrgw01).
- Логика самой функции — live smoke-тест на arch03 по той же схеме, что уже применялась для `ceph_watchdog_recover()` (sed-экстракция функций, синтетический вызов): искусственно создать запись в blocklist для IP хоста (`ceph osd blocklist add <host-ip>:0/0` или похожий синтетический адрес), убедиться что новый блок её находит и снимает, и что дальнейший `mount` проходит.
- Проверить путь "записей нет" (blocklist пуст) — убедиться, что функция ведёт себя как раньше (переходит к обычному `return 1`), не пытается ничего снять и не ломается на пустом списке.
