# Мониторинг бэкапов arch03/04/05 — дизайн

Дата: 2026-07-07
Статус: утверждён пользователем, готов к написанию плана реализации

## Цель

Информативная, наглядная веб-страница на русском языке, показывающая в реальном времени
статус резервного копирования на трёх хостах (arch03, arch04, arch05): успешность
последнего прогона, текущий прогресс (если бэкап идёт сейчас), состояние Ceph-mount,
ошибки. Страница должна обновляться автономно (без участия человека и без открытой
сессии Claude), с интервалом в единицы минут.

## Контекст и ограничение, определившее архитектуру

Единственный путь к arch03/04/05 — `amar319 → wn75 → arch0X` (см. память
`ssh_topology_arch_backup.md`). Личный ключ пользователя на amar319
(`~/.ssh/id_ed25519`) защищён паролем и требует ручного ввода passphrase каждые ~10
минут (ControlMaster истекает) — на этом ключе нельзя построить автономный сбор данных.
Решение: завести **отдельный ключ без пароля**, но с жёстким ограничением через
`command=` в `authorized_keys`, чтобы даже при компрометации ключа с ним нельзя было
выполнить ничего, кроме чтения статуса.

Полная интеграция в существующую Grafana (http://144.206.234.174:3001, Prometheus
подтверждён — node_exporter уже работает на arch0X, порт 9100) была рассмотрена, но
отложена: сеть ceph-хостов не имеет маршрута к Grafana/Prometheus хосту, а найти адрес
Prometheus через UI Grafana не удалось в разумное время. Отдельная страница — это
рабочее решение на сейчас; интеграция в Grafana остаётся возможным будущим шагом,
когда доступ и топология сети будут прояснены.

## Архитектура

```
[cron на arch03]     [cron на arch04]     [cron на arch05]
  backup_status.sh -> backup_status.sh -> backup_status.sh
       |                    |                    |
  /var/lib/backup-status/status.json (локально, без сети)
       |                    |                    |
       +--------------------+--------------------+
                            |
              (раз в ~2 мин, по ограниченному ключу,
               command="...backup_status.sh" в authorized_keys)
                            v
              [cron на amar319: collect_status.sh]
                            |
                            v
          ~/backup-monitor/status.json (агрегированный)
                            |
                            v
        [локальный HTTP-сервер на amar319, 127.0.0.1:8077]
                            |
                            v
           dashboard.html (auto-refresh раз в 30-60с,
                fetch('status.json'), рендер в браузере)
```

## Компоненты

### 1. Ключ мониторинга

- Новая пара `ssh-keygen -t ed25519 -N "" -f ~/.ssh/monitor_ed25519` на amar319, без
  passphrase.
- Публичный ключ добавляется в `authorized_keys`:
  - На **wn75**: обычная запись (без ограничения — это просто транзитный прыжок для
    ProxyJump, `-W` форвардинг).
  - На **arch03/04/05** (root): с форсированной командой:
    ```
    command="/usr/local/bin/scripts/backup_status.sh --print",no-port-forwarding,no-X11-forwarding,no-agent-forwarding,no-pty ssh-ed25519 AAAA... monitor@amar319
    ```
    Независимо от того, что клиент запросит выполнить, сервер всегда выполнит именно
    этот скрипт. Ключ физически не может открыть интерактивный shell или выполнить
    произвольную команду.
- Это изменение вносится **только с явного согласия пользователя** на каждом хосте
  (per [[feedback_no-unilateral-infra-changes]]) — план реализации должен показать
  точные команды перед выполнением.

### 2. `backup_status.sh` (новый файл, деплоится на каждый arch0X)

Путь: `/usr/local/bin/scripts/backup_status.sh`. Запускается двумя способами:
- По cron каждые 2 минуты (пишет файл локально) — **основной путь**, не зависит от
  того, дёрнули ли его по SSH.
- Как forced-command по SSH (просто выводит содержимое уже готового файла в stdout —
  дешёвая операция, не пересчитывает статус на лету, чтобы SSH-запрос не мог случайно
  наткнуться на дорогую операцию типа `find` по CephFS).

Логика генерации (cron-режим), пишет `/var/lib/backup-status/status.json`:

```json
{
  "host": "arch03",
  "generated_at": "2026-07-07T10:15:00+03:00",
  "last_success": {
    "finished_at": "2026-07-07T05:00:21+03:00",
    "files_copied": 0,
    "files_deleted": 0,
    "errors": 0,
    "duration_sec": 8579
  },
  "running_now": {
    "active": true,
    "started_at": "2026-07-07T07:36:47+03:00",
    "checks_done": 7963,
    "checks_total": 17979,
    "percent": 44
  },
  "ceph": {
    "mounted": true,
    "accessible": true,
    "last_mds_incident": "2026-07-07T05:02:04+03:00"
  },
  "disk": {
    "backup_used_percent": 72,
    "backup_avail_human": "35T"
  }
}
```

Источники данных внутри скрипта:
- `last_success`: последний `*.summary.json` в `/var/log/backup/` (см.
  [[backup-log-locations]]).
- `running_now`: `systemctl is-active rclone-backup.service` — если `active`/
  `activating`, парсит последнюю строку `Checks: X / Y` из самого свежего
  `backup_*.log`.
- `ceph`: `mount | grep -q ceph` для факта монтирования, плюс `timeout 3 stat /ceph`
  для реальной доступности (ловит именно случай `mds0 rejected session`, когда mount
  формально есть, но обращение виснет/даёт Permission denied — см.
  [[ceph-mount-dropout-incidents]]). Время последнего инцидента — из
  `dmesg -T | grep -i "rejected session"` (последняя строка, если есть).
- `disk`: `df -h /backup`.

Если какой-то источник недоступен (например, нет ни одного `.summary.json`), поле
просто становится `null`, скрипт не падает.

### 3. `collect_status.sh` (новый файл на amar319)

По cron каждые 2 минуты: для каждого хоста (arch03, arch04, arch05) выполняет
`ssh -i ~/.ssh/monitor_ed25519 -o ConnectTimeout=8 -o BatchMode=yes arch0X` (ProxyJump
через wn75 уже в `~/.ssh/config`) — сервер сам подставит forced command, что бы ни
было передано. Читает JSON из stdout, складывает в
`~/backup-monitor/status.json`:

```json
{
  "generated_at": "2026-07-07T10:16:03+03:00",
  "hosts": {
    "arch03": { ...последний успешно полученный JSON..., "stale": false },
    "arch04": { ..., "stale": false },
    "arch05": { ..., "stale": true, "last_seen": "2026-07-07T09:40:00+03:00" }
  }
}
```

Если SSH к какому-то хосту не удался (сеть/ceph легли, wn75 недоступен) —
`collect_status.sh` **не затирает** предыдущие данные по этому хосту, а помечает их
`"stale": true` и оставляет `last_seen` от последнего успешного сбора. Так дашборд
всегда может показать «последнее известное состояние», а не просто пропасть.

### 4. Веб-страница

- Простой HTTP-сервер: `python3 -m http.server 8077 --bind 127.0.0.1` (или `--bind`
  на LAN-адрес amar319, если нужно открывать не только с самого amar319 — уточняется в
  плане реализации), поднятый через systemd user-сервис или cron `@reboot`, отдаёт
  директорию `~/backup-monitor/` (там же лежит `status.json` и `dashboard.html`).
- `dashboard.html` — статический файл, ванильный HTML/CSS/JS (без сборки, без внешних
  CDN — сервер локальный, интернета может не быть):
  - Карточка на каждый хост: цветной индикатор (🟢/🟡/🔴), название хоста, время с
    последнего успешного бэкапа человеко-читаемо («3 часа назад»), файлов
    скопировано/удалено/ошибок из последнего прогона.
  - Если `running_now.active` — горизонтальный прогресс-бар с процентом и числами
    (checks_done/checks_total).
  - Полоска состояния Ceph (зелёная/красная), с временем последнего инцидента если
    был.
  - Если `stale: true` — карточка визуально приглушена (серая рамка) с пометкой «нет
    свежих данных с {last_seen}».
  - Мини-история: последние 14 результатов бэкапа на хост в виде ряда маленьких
    цветных квадратиков (успех/ошибка/пропуск) — данные копятся в `status.json`
    построчно при каждом сборе (rolling window, не отдельный файл).
  - Автообновление: `setInterval` раз в 30 секунд делает `fetch('status.json')` и
    перерисовывает DOM — без перезагрузки страницы.

**Пороги предупреждений:**
- 🟢 зелёный: последний успешный бэкап < 30 часов назад, Ceph доступен, 0 реальных
  ошибок.
- 🟡 жёлтый: последний успешный бэкап 30–48 часов назад, ИЛИ есть ошибки в последнем
  прогоне, ИЛИ данные `stale`.
- 🔴 красный: последнего успешного бэкапа > 48 часов, ИЛИ Ceph сейчас недоступен
  (`accessible: false`).

## Безопасность

- Новый ключ — без пароля, но `command=`-ограничен на arch0X до одного конкретного
  read-only скрипта; на wn75 — обычный (проброс, без выполнения команд на самом wn75).
- `backup_status.sh` не принимает никаких аргументов от SSH-клиента (даже если бы
  клиент попытался их передать, forced command их игнорирует) — исключает инъекцию
  команд через SSH command string.
- Веб-сервер слушает только `127.0.0.1` (или LAN, но не публичный интернет) — решение
  подтверждено пользователем.
- Никаких секретов (паролей, приватных ключей) в `status.json`/`dashboard.html` не
  публикуется.

## Тестирование

- Ручной запуск `backup_status.sh` на каждом arch0X — проверить, что JSON валиден и
  соответствует реальному состоянию (сверить с уже виденными в этой сессии данными:
  последний `.summary.json`, реальный статус ceph mount).
- Ручной запуск `collect_status.sh` на amar319 — проверить агрегацию и обработку
  `stale` (например, временно заблокировать доступ к одному хосту и убедиться, что
  старые данные сохраняются, а не исчезают).
- Открыть `dashboard.html` в браузере, убедиться что автообновление работает и
  пороги (🟢/🟡/🔴) корректно вычисляются на реальных и искусственно испорченных данных.

## Вне рамок (возможное будущее развитие)

- Интеграция в существующую Grafana (после того как будет найден адрес Prometheus и
  прояснён сетевой доступ от ceph-хостов).
- Уведомления (Telegram/email) при переходе хоста в 🔴 — не запрошено сейчас, но
  архитектура (отдельный `status.json`) позволяет добавить это позже как ещё один
  маленький скрипт, читающий тот же файл.
- Починка самой обнаруженной проблемы «бэкап молча теряет файлы при нестабильном
  Ceph» (см. [[ceph-mount-dropout-incidents]]) — отдельная задача, не входит в этот
  дашборд.
