# Дашборд: системные метрики хоста — дизайн

**Дата:** 2026-07-08
**Статус:** одобрено пользователем

## Цель

Дашборд мониторинга бэкапов (`monitoring/dashboard.html`) сейчас показывает статус самого бэкапа (последний успешный прогон, прогресс, здоровье Ceph, место на диске), но не показывает состояние самого хоста. Нужно добавить то, что обычно смотрят через `top`/`htop`: load average, использование памяти, аптайм и количество активных процессов `rclone`. Возможны дополнительные метрики в будущем.

## Архитектура

Существующий поток данных не меняется, только расширяется:

```
arch0X: backup_status.sh (cron, каждые 2 мин) → /var/lib/backup-status/status.json
                                                          ↓
amar319: collect_status.sh (timer, каждые 2 мин) → SSH forced-command (monitor_ed25519) → status.json --print
                                                          ↓
                                          _merge_status.py → ~/backup-monitor/www/status.json
                                                          ↓
                                          dashboard.html (браузер, автообновление 30с)
```

Новые системные метрики собираются в том же скрипте (`backup_status.sh`), в том же 2-минутном цикле, и передаются по тому же пути — никакой новой инфраструктуры (нового cron/timer/forced-command) не добавляется.

## Сбор данных (backup_status.sh)

Новый блок перед сборкой итогового JSON, использует только `/proc` (без зависимости от `top`/`vmstat`, чей вывод не всегда одинаков между версиями util-linux на разных хостах — уже сталкивались с этим на старых системах):

- **Load average** — `/proc/loadavg`, первые 3 поля (1/5/15 мин), как есть, без порогов/интерпретации.
- **Память** — `/proc/meminfo`: `MemTotal` и `MemAvailable` (не `MemFree` — `MemAvailable` учитывает вытесняемый кэш/буферы и честнее отражает реальную доступную память). `used_gb = (total - available) / 1024^2`, `total_gb = total / 1024^2`, `percent = round(used/total*100)`.
- **Аптайм** — `/proc/uptime`, первое число (секунды с загрузки). Пересчитывается в `boot_at` — ISO-8601 timestamp момента загрузки (`generated_at − uptime_seconds`), в том же формате (`date -Iseconds`), что и остальные timestamp'ы в JSON. Форматирование "N дней/часов назад" делает дашборд на клиенте, переиспользуя существующий JS-хелпер `humanAgo()`.
- **Процессы rclone** — `pgrep -c -x rclone` (точное совпадение имени процесса — та же конвенция, что `pkill -x rclone` в ceph_watchdog, чтобы не подхватить процесс, вызывающий сам pgrep). Если процессов нет, `pgrep` возвращает ненулевой код — обрабатывается как `0`, не как ошибка.

## Схема JSON

Новый ключ `system` на уровне объекта хоста, рядом с существующим `disk`:

```json
"system": {
  "load_avg": [0.52, 0.61, 0.58],
  "memory": {"used_gb": 12.3, "total_gb": 64.0, "percent": 19},
  "boot_at": "2026-01-01T03:00:00+03:00",
  "rclone_processes": 4
}
```

Плоский объект — добавление новых метрик в будущем означает просто новые ключи здесь, без изменения структуры.

## Обработка ошибок

Если сбор какой-то из системных метрик не удался (например `/proc/meminfo` недоступен) — весь блок `system` становится `null` для этого запуска, скрипт не падает. Это тот же паттерн, что уже используется для `last_success` (см. `backup_status.sh`). Дашборд должен трактовать `system: null` как "нет данных" (аналогично уже существующей обработке `ceph: null`/`cephUnknown`), не как ошибку.

## Дашборд (dashboard.html)

Новый блок на карточке хоста, под существующей секцией Ceph:
- `Load average: 0.52 / 0.61 / 0.58`
- `Память: 12.3 / 64 ГБ (19%)`
- `Аптайм: <результат humanAgo(boot_at) без суффикса "назад">`
- `Процессы rclone: 4`

Без цветовых порогов/предупреждений — чисто информационный блок, как принято в текущей версии.

- Симулировать отсутствие `/proc/meminfo` (или испортить парсинг) — убедиться, что скрипт не падает, весь `system` становится `null`, остальной JSON генерируется как обычно.
- `dashboard.html`: открыть в браузере, проверить рендер блока для хоста с `system` и для случая `system: null`.
